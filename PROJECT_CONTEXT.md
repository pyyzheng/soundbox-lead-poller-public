## 二、架构意图

### 2.1 为什么这样设计

- **云端轮询替代本地 cron**：本地 cron 依赖 Mac 在线，休眠/断网即停止。GitHub Actions 24/7 运行，免运维。过渡期保留本地 cron 只读模式作为兜底监控。
- **Gmail label 做幂等控制**：每封处理过的邮件打上 `processed-by-openclaw` 标签，搜索时排除已处理邮件，天然去重，无需外部数据库。
- **三档过滤（pass/review/reject）**：二元过滤会误杀单信号线索（如短消息真人询盘）。review 档带 `[待确认]` 前缀写入飞书，让业务员判断，避免漏单。
- **智谱 API 做 LLM 解析**：免费额度充足，中文和多语言解析能力满足需求，响应速度快。
- **飞书作为数据后端**：团队日常协作已在飞书，多维表格支持自动化视图、提醒和内置分配规则，业务员直接消费数据。

### 2.2 架构决策记录（ADR）

| 编号 | 决策 | 原因 | 替代方案及弃用理由 | 日期 |
|------|------|------|-------------------|------|
| ADR-001 | GitHub Actions 而非 VPS | 免运维、与代码仓库天然集成 | VPS：需维护；本地 cron：Mac 关机即停 | 2026-04-18 |
| ADR-002 | Gmail label 做幂等控制 | 原子操作、无需外部状态数据库 | 数据库去重：引入额外依赖和一致性问题 | 2026-04-18 |
| ADR-003 | 三档过滤 pass/review/reject | 单信号线索不应直接丢弃，review 档避免误杀 | 二元过滤：会丢失边界线索 | 2026-04-18 |
| ADR-004 | 去重升级到 email + message 前 80 字符 | 允许回头客二次询盘通过 | 纯 email 匹配：误拦回头客新询盘 | 2026-04-18 |
| ADR-005 | 智谱 API 做 LLM 解析 | 免费额度充足、中文解析能力足够 | OpenAI：成本高；本地模型：准确度不足 | 2026-04-18 |
| ADR-006 | 本地 cron 只读 + 云端写入 | 过渡期消除双写冲突 | 双写：飞书出现重复记录 | 2026-04-20 |
| ADR-007 | 舱网子渠道从 subject 匹配 | 四个网站共用 service@soundbox-sys.com，无法从发件人区分 | 邮件 body 解析：不稳定、HTML 干扰 | 2026-04-20 |

## 三、正常状态定义

### 3.1 关键指标

| 指标 | 正常范围 | 检测方式 | 异常时的影响 |
|------|---------|---------|-------------|
| 日均线索量 | 10-30 条 | 飞书每日报告 | 偏离过大 → 可能有漏单或垃圾涌入 |
| 每轮处理邮件数 | 0-5 封 | Actions 日志 | 持续 >10 → 搜索范围异常 |
| 处理延迟 | ≤30 分钟（轮询间隔） | Gmail 时间戳 vs 飞书记录时间 | 业务员响应慢 |
| 垃圾过滤率 | ≥90% | 健康检查重验飞书记录 | 业务员被垃圾信息干扰 |
| Actions 成功率 | ≥95%（48 次/天） | GitHub API | 轮询通道静默失效 |
| Gmail API 调用 | ≤10 次/轮 | Actions 日志 | 超限可能被限流 |
| OAuth token | 有效 | health-check 每 6h 验证 | 管道完全停摆 |

### 3.2 系统不变量

1. 每封被标记 `processed-by-openclaw` 的邮件，在飞书表中必有且仅有一条对应记录（或被过滤链 reject）
2. 同一封邮件不得产生多条飞书记录（去重保证）
3. reject 的邮件必须被打上 Gmail 标签（确保下次搜索不再命中）
4. 飞书记录的 `Enquiry details` 字段不得为空（必须有原始询盘内容）
5. 所有网络请求必须包含超时设置（≤30 秒）

## 五、护栏规则

### 5.1 过滤与校验规则

→ **详见 [QA-RULES.md](QA-RULES.md)**，包含：6 层过滤链架构、垃圾邮件特征库、阈值配置、回归测试用例。

### 5.2 业务规则

1. 线索写入飞书后，由飞书内置规则自动完成业务员分配，代码不涉及分配逻辑
2. review 档线索带 `[待确认-疑似垃圾/内容不明]` 前缀写入飞书，由业务员人工判断
3. reject 的邮件只打 Gmail 标签，不写入飞书，不丢弃原始邮件
4. 回头客二次询盘（同 email、不同 message）应正常写入，不被去重拦截
5. `lead-rules.json` 是唯一规则配置源，修改时必须同步 QA-RULES.md 中的阈值表
6. 舱网系列（新官网/总舱网/加拿大舱网/美国舱网）从 `service@soundbox-sys.com` 发来，子渠道通过邮件 subject 关键词识别（新官网/总舱网/加拿大舱网/美国舱网），字段格式见 QA-RULES.md

### 5.3 代码规范

1. 所有网络请求必须包含超时设置（≤30 秒）
2. 不得硬编码飞书 App ID/Secret，必须使用环境变量或 GitHub Secrets
3. `cloud-*.py` 脚本不使用代理（GitHub Actions 直连）；本地脚本通过 `HTTPS_PROXY=http://127.0.0.1:7890`
4. 新增文件建议不超过 500 行；超过时确保函数职责单一。cloud-lead-poller.py 作为主管道入口例外，新功能优先放 lib/

### 5.4 禁止事项

1. 永远不要删除已分配的飞书记录（标记为垃圾即可）
2. 永远不要在没有 email + message 去重的情况下写入飞书
3. 永远不要跳过 Gmail label 标记步骤
4. 永远不要在代码中硬编码任何 API key 或 secret
5. 修改过滤规则后必须跑 QA-RULES.md 第五节回归用例

## 七、人类治理节奏

### 每天（晨间）

- 查看飞书中是否有健康检查上报的异常通知
- 如有异常：处理 → 补充异常模式库（第四节）

### 每周（30 分钟"垃圾回收"）

- 回顾本周所有亲自介入的问题
- 问：哪些可以变成规则写进第五节？哪些可以变成检测逻辑加进 health-check？
- 更新本文档的"最后更新"日期
- 检查异常模式库中"自动检测"列，把"否"变成"是"

### 每月（1 小时）

- 审视架构决策记录（第二节），是否有需要修订的过时决策
- 评估系统整体健康趋势：异常频率是否在下降？
- 考虑是否有新的不变量需要添加（第三节）
